# 信任降級重設計 — 徵詢 Claude Design(#2)

> 背景:habit-rank 家庭自律 app。信任系統目前的「退回一次掉一整級」是斷崖式懲罰,要重設計。
> 這份是實作方(Claude Code)依「行為設計顧問」角色做的診斷 + 提案 + 推薦,請你審、補漏、或提更好的做法。
> 既有負面清單持續有效:不做手足比較、不碰隨機/賭博、不增加螢幕黏著、獎勵準時停止而非任務量。

---

## 0. 現況與 bug

信任 per-behavior,`trustLevel` 0/1/2(每次確認 / 30% 抽查 / 畢業免驗證)。
升級:連續誠實 `trustPromoteDays[14,21]` 天 + 達成率地板 0.6。
**降級(問題所在)**:家長退回一筆打卡 → `trustLevel = max(0, lvl-1)`,而且退回事件 `verdict='rejected'` 同時打斷「誠實回報連續」。

→ 一次退回 = **雙重懲罰**:退一整級(14–21 天心血)+ 斷連續。
兩個病灶:
1. **破戒效應(abstinence violation effect)**:單次失誤被框成「整個破功」→ 自暴自棄。
2. **根因**:`trustLevel` 是被直接 mutate 的可變欄位,不是從 append-only 事件流推導——「退一級」這種粗暴操作正是因為它是可變欄位才做得出來。
3. 家長「按錯退回」(孩子其實有做)目前跟「說謊被抓」一模一樣處理 → 冤枉,且無更正機制。

---

## 1. 硬約束(已定案,方案必須全部滿足)

1. **誠實回報失敗的信任損失 < 說謊被抓的信任損失**,這個不等式在任何情境下都要成立——誠實才是孩子的理性策略。
2. 區分「失誤(lapse)」與「刻意說謊(lie)」,兩者處理必須不同。
3. 單次失誤的信任損失 **不得超過 3 天進度**,**禁止整級退回**。
4. 附狀態機:信任分數在「誠實達成 / 誠實回報失敗 / 說謊被抓 / 家長記錯誤退」四種事件下各如何變化。

---

## 2. 方案 A(實作方推薦):信任 = 連續分數 T,由 append-only 事件流「推導」,不存可變欄位

**心理機制**
- 用**累積分數**取代**連續天數**,落地 Lally 習慣自動化曲線(「偶爾漏一次不影響最終自動化」)。連續是全有全無、一斷歸零;累積天生有緩衝。
- lapse 與 lie 用不同 ΔT 明確分流(Marlatt)。
- 誠實的獎勵放 **XP 軸**(現行 +5 XP 不變),信任軸對誠實回報**中性**——兩軸不互兌。

**狀態機**(每個行為一條信任分數 `T`,單位 = 天當量;等級由門檻推導)

```
狀態:T ≥ 0(每個行為一條)
等級 L = 0 若 T<14 · 1 若 14≤T<35 · 2 若 T≥35   (L2 = 畢業免驗證)
確認強度依 L:L0 每次確認 · L1 30% 抽查 · L2 即時入帳

事件 → ΔT(固定值,任何情境不變):
①  誠實達成    done + 通過/抽查過/畢業即時      ΔT = +1
②  誠實回報失敗 markMiss(honest)                ΔT =  0   ← 誠實不扣分(XP 另給 +5)
③  說謊被抓    claimed done → 家長退回          ΔT = −5   ← 5 個誠實達成日補回,不退級
④  家長記錯誤退 退回但孩子其實有做              ΔT = −5 暫扣 → 家長「撤回退回」或孩子申訴成立 → 補回 +5(淨 0)

邊界:T 夾在 [0, ∞);單次最大跌幅 = 5 < 14(一級)⇒ 結構上不可能整級退回
```

**四約束對照**:①`0 < 5` 常數恆成立 ✓ ②lapse 0 vs lie −5 明確不同 ✓ ③失誤損失 0 ≤ 3、最大跌幅 5 < 14 ✓ ④四事件如上 ✓

**資料結構影響**
- 移除 `trustLevel` 直接 mutate;改 `trustScoreOf(events, behaviorId)` 純函式:掃該行為「每天最新事件」累加 ΔT;等級 = `T→L` 門檻函式。
- 「④ 撤回退回」= **append 一筆更正事件**(當日 `verdict='approved'`),因 `todayEventOf` 取最新,自動蓋掉 rejected → 純 append-only、零 mutate。
- `graduatedAt` 保留;移轉:舊 `trustLevel` 依門檻回填 T 基線(L1→14、L2→35)。
- **待解實作細節**:事件目前只留 60 天,T 純推導時舊事件老化會讓 T 漂移。解法:trust 相關事件延長保留,或每次 rollover 存 T checkpoint。動工前要定案。

**反向指標**:某行為升上 L2(即時入帳)後,家長偶測發現實際達成率驟降 → 在 p≈0 區被鑽漏。

**最大風險**:把信任從「存欄位」改「推導」是動地基;T checkpoint / 事件保留沒處理好,可能出現「信任莫名下滑」的詭異 bug,比斷崖更難 debug。

---

## 3. 方案 B(最小改動備案):保留 `trustLevel`,退回只扣「級內進度」不退級

- 每級內另存進度 `prog`(天);升級看 `prog≥門檻`;退回 `prog -= 5`(夾 0),歸零也不掉級;lapse 不動 `prog`。
- 資料:新增 `trustProg[bid]`,`trustLevel` 只增不減。
- **缺點**:仍是 mutate 欄位 → 不符 append-only;未來上 Supabase 多裝置同步容易 race、對不齊。
- **最大風險**:緩坡取代斷崖,但地基照樣違反 append-only,根因沒解。

---

## 4. 推薦

**推薦方案 A。** 它同時修掉斷崖 + 根因(可變欄位違反 append-only);累積分數落地 Lally、與已上線的「彈性連續(滾動 7 天)」同一套哲學;「撤回退回=append 更正事件」乾淨解第 4 種事件;事件推導的信任天生適合接下來的埋點(#3)與 Supabase 同步(單一真相=事件流)。B 只在「這週就要上、不想動地基」時選。

---

## 5. 給 Claude Design 拍板的問題

1. **A vs B**:同意走 A(事件推導、動地基)嗎?還是有理由先上 B?
2. **ΔT 數值**:說謊 −5、失誤 0、達成 +1——這組啟發式合理嗎?−5 會不會讓孩子只打穩贏的卡、迴避有風險任務?建議區間?
3. **第 4 種事件**:家長「撤回退回」(家長端一鍵更正)vs 孩子「我有做到,請再看」(孩子申訴回家長複審)——先做哪個?兩個都做?哪個比較不傷孩子自主感又能自我更正?
4. **達成率地板**:A 裡 T 只由「真的做到」累積,達成率地板 0.6 是否可移除(已內含),還是保留當次要保險?
5. **低 catch-rate 漏洞**:L2 即時入帳(p≈0)時,未被抓的說謊仍 +1。這是「信任遞減驗證」的設計代價,接受嗎?還是要保留某種隨機偶測(注意:不能變成賭博機制)?
