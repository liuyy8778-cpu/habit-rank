## 🔒 安全 / 技術債

    前端陣列。append-only 只存在於前端 state,DB 層不是真 append-only——若 insert 中途失敗/斷網,該小孩
    的打卡歷史會整段消失。
  - 對照:商店的 `ledger_events`(2026-07 新增)已是**真 append-only**(只 insert、無 UPDATE/DELETE
    policy、購買走 `purchase_item` RPC 原子交易)。checkin_events 應比照重構為**增量同步**
    (只 insert 新事件 / upsert by 穩定 id,永不整段 delete)。

  (Supabase RLS 依登入身分限制家長資料存取)。與上面 PIN RPC 同批未來處理。

## 🗳️ 提案節流規格【B】(入規格,與 Phase 2 提案功能同批)

1. **送出時間**:永遠開放,不設每日次數上限(靈感不該被營業時間丟棄)。
2. **待審上限(佇列節流,非配額拒絕)**:任務提案同時最多 2 件 pending;**獎品提案獨立最多 2 件**;條款/公約提案最多 1 件。達上限提示「先等爸媽審完排隊中的提案」——逼孩子自己排優先序。

5. **理由必填**:核准/拒絕都留一句家長理由(存提案歷史)——判例集,孩子下次提案的地圖。

## 🛒 商城功能規格(暫存,真機驗證通過後接續開發)

### 共享型商品規則(design-advisor 經濟章 + 商城落地;規格已定,待實作)
- **定義**:凡「全家共同參與」的商品(看電影、外出用餐、家庭活動),購買者買到的是**發起權 + 決定權**(這件事因我而發生、細節我說了算),**不是獨佔權**——其他家人照常參與,不需分攤、不需配合付費。
- **原則**:① 禁止要求手足分攤共享商品費用(接「購買是純選擇權」鐵律)② 手足不平衡感導向貨架:「同一件商品,你也買得起,同一個價」(同物同價鐵律應用)——羨慕轉化為儲蓄動機,不走分贓/輪流補償。
- **功能落地**:
  1. 商品可標記「**共享型**」;說明必須寫明買到的權利範圍(例:「全家一起去,你決定看哪部、哪一天」)——購前零模糊、兌現零爭議。
  2. **排程條款**:共享型說明加註「日期需選全家可行的時間」(發起權不是綁架權)。
  3. **家庭額度**:消耗家長時間的共享商品可設「每月全家額度 X 次」為**公開商品屬性**(如電影每月 2 次),用罄當月不可購、下月重置。需 family 層級購買計數(跨小孩共用額度)——建議 covenant.purchase_log jsonb({itemId,kidId,ym,at}),與 custom_shop 同期上雲。
  4. 資料:customShop 加 `shared`(bool)、`monthlyQuota`(int|null);shared+quota 商品改可重複購(非一次性 redeemed),受額度上限。
- **家長話術**(備忘,非程式):「他買的是『這次聽他的』,不是『你不能來』——你想要『聽你的』,貨架上同一個價。」

### 現金類特殊商品型別(登記,待真實使用一輪後定規格,不急)
- 現金 / 零用錢類商品:匯率偏貴(現金稀釋 coin 的體驗價值)+ 週上限。暫時人工把關(改道到 reward、手動議價 + 段位鎖)。規格待實測後定案再開發。

## 📟 時段型 NFC 打卡「兩嗶制」(第二波;單嗶交機動線跑順後評估)

同一 token 兩次感應構成一段「在場休息」區間。第一波只做單嗶交機打卡,先跑順再評估這個。
1. **新事件對**:`rest_start` / `rest_end`(同一 token:第一嗶=start、第二嗶=end)。
2. **成立條件**:end − start ≥ 任務定義時長(如 30 分)→ 寫入該任務完成事件;不足時長 → 顯示「還差 X 分鐘,繼續休息」,不發獎勵、start 保留可續。
3. **當日僅計一次有效區間**;跨日自動作廢。
4. **防呆**:已有未關閉的 start 時再嗶 → 視為 end(不疊加開多個 session)。
5. **不偵測中途拿起手機**(技術上不可行且不必要)——由誠實軸 + 物理在場接住,同交機哲學。
6. **孩子端文案**:「嗶一下,手機面朝下放好,去休息。回來再嗶一下領獎勵。」
> 依賴:第一波單嗶 NFC 路由(/checkin?token) + source 標記先落地;兩嗶制在其資料模型上擴充。
