# 多個小孩 — 設計簡報(徵詢 Claude Design)

> 現況:app 是「單一小孩」設計,所有進度存成一份。家庭有 2 個(以上)小孩時要怎麼處理?
> 這份文件整理問題、我的建議、以及需要拍板的決策點。

---

## 0. 現況與問題

- 目前每個小孩的資料(金幣、XP、段位、信任等級、連續、打卡事件、taskOn)全部存成**一份扁平 state**,放在該瀏覽器的 localStorage。
- 共享/家庭層級的東西:家庭公約、任務庫、獎勵商店、家長。
- 兩個小孩 → 需要:每個小孩各自一份進度、家長要能確認兩個小孩、而且**絕對不能變成手足比較**。

---

## 1. 核心設計原則(延續既有理念)

1. **絕不做手足排行榜 / 並排比較**(這是既有的設計負面清單)。每個小孩的段位、信任只跟**自己的過去**比。家長看得到兩個小孩,但 app 不呈現「誰贏誰」。
2. **各自獨立**:段位、信任、金幣、連續、打卡——每個小孩完全獨立。
3. **家庭層級共享**:公約、全家 BOSS 戰(合計、不比較個人)、獎勵 catalog(可共享模板)。
4. 家長端仍守 **30 秒原則**:兩個小孩的待確認也要能快速處理。

---

## 2. 我建議的資料模型

從「單一扁平 state」改成「家庭 + 成員」:

```js
family: {
  covenant: {...},          // 共享:公約(可含每個小孩各自的交機時間)
  rewards: [...],           // 獎勵 catalog(共享模板)
  parent: { name },
  kids: {
    kidA: {
      name, avatar, age,
      coins, xp, streak,
      taskOn, manualUnlock, schedules,      // 各自的啟用任務、時段
      trustLevel, graduatedAt,              // 各自的信任線
    },
    kidB: { ... },
  },
}
checkinEvents: [ { kidId, behaviorId, ... } ]   // 每筆打卡標記是哪個小孩
currentKid: 'kidA'                               // 目前檢視的是哪個小孩
```

- 每筆 `checkinEvent` 加一個 `kidId`。
- 家長待確認 = 依 `kidId` 分組/過濾。

---

## 3. UI 的關鍵改變

- **頂部切換**:現在是「小孩 / 家長」→ 改成「小孩A / 小孩B / 家長」(或小孩下拉選)。
- **小孩視角**:只顯示自己的資料;切到另一個小孩要能簡單切(共用裝置時)。
- **家長待確認**:兩個小孩的待確認,用**每個小孩一個分頁 or 標籤**呈現(每筆標明是誰),一鍵全過可分別處理。
- **週報 / 段位 / 信任**:各自一份,分開看。
- **公約**:一份家庭公約,但**交機時間可以每個小孩不同**(大的晚一點、小的早一點)。

---

## 4. 每個小孩可以不一樣(客製化)

不同年齡需求差很多,所以這些**各自獨立**:
- 起步任務 / 啟用任務(taskOn)
- 交機時間(schedules)
- 段位、信任、金幣進度

**只有**公約條款、全家 BOSS 戰、獎勵 catalog 是共享的。

---

## 5. 我的核心建議:跟「帳號 / 後端」一起做

**「多個小孩」= 「家庭有 N 個成員」= 帳號 / 後端(Supabase 家庭空間)本來就要解的問題。** 所以:

- **推薦**:把多小孩**併進帳號/後端那一階段**設計。家庭空間 → 家長帳號 → N 個小孩 profile → 每台裝置選(或綁定)一個 profile。這樣「共用裝置」和「各自裝置」都自然涵蓋,而且雲端同步。
- **不推薦**:現在硬把 localStorage 單一 blob 改成多小孩(可行但會做兩次工——之後上後端還要再改一次)。
- **折衷(如果現在就非要能用)**:先在 localStorage 存 `kids` map + 一個小孩切換器,當作 interim;之後上後端時資料結構已對齊、好搬。

---

## 6. 需要 Claude Design 拍板的決策

1. **裝置模式**:兩個小孩**共用一台**(家庭平板)還是**各自有裝置**?(決定要 in-app 切換,還是靠帳號綁定)
2. **時機**:多小孩**跟帳號/後端一起做**(我推薦),還是**先做 localStorage interim**?
3. **反手足比較**:確認「每個小孩只看自己、家長看得到全部但 app 不排名」——同意嗎?有沒有想保留某種良性的家庭共同進度(BOSS 戰)?
4. **客製化範圍**:起步任務 / 交機時間 / 段位進度全部各自獨立、只有公約+BOSS+獎勵共享——這個切法對嗎?
5. **家長待確認的呈現**:兩個小孩的待確認要「每人一分頁」、「一個混合列表(每筆標名字)」、還是「切換看單一小孩」?
6. **獎勵商店**:共用一份 catalog、但各自用各自的金幣兌換?還是每個小孩各自 catalog?

---

## 7. 給實作者的備註(我自己)

- 這是**大改**:整個 state 從扁平變成 per-kid,幾乎每個讀 coins/xp/habit/checkinEvents/trustLevel 的地方都要 scope 到 currentKid。
- 若走後端,per-kid 天生就是「多列 / 多 profile」,比 localStorage 巢狀 map 乾淨。
- 遷移:現有單一小孩資料 → 變成 kids.kidA(第一個小孩),無損。
