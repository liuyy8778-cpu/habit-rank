## 2. 技術限制(這幾點會影響 2.6 / 2.5 的做法,請知悉)

我實際查過這個框架(Claude Design 匯出的 x-dc + React runtime),兩個要注意:

1. **目前整個 app 一個 `<input>`/`<canvas>` 都沒用過** —— 表單與 canvas 是全新未測領域。
   框架有支援 `onInput` / `onChange`,理論上控制型輸入可行,但我會**先做一個輸入 spike 驗證**再蓋公約頁。

2. **框架的事件表沒有 `mousemove` / `touchmove`**(只有 down/up/enter/leave)。
   → 這代表**手寫簽名 canvas 不能用模板事件綁定**,得在 JS 裡手動 `addEventListener`。**可行但費工。**

**影響 2.6 公約頁的簽名:** 見第 5 節 Q3,我提議先出「打字姓名版」簽名。

## 3. 我的設計補強提案(想聽你 agree / 不 agree)

### A.（對應 2.4 / 2.2)「誠實回報鏈」有反刷分的破口

文件設計:小孩每天**誠實承認「沒做到」也不斷連續、還拿 0.5× XP**。理念我完全認同(獎勵誠實而非完美)。但兩個實務隱憂:

- **可刷分**:每天誠實擺爛也能養連續 + 領 0.5× XP,行為零改變卻穩定進帳。
- **家長困惑**:看到「連續 30 天」但實際很少做到,會誤解。

**我的提案**(擇一,見 Q1):
- (i) honest-miss **保連續、但 XP 給 0 或象徵性**(不是 0.5×);或
- (ii) **連續 honest-miss 遞減/封頂**(如連 3 天沒做到,連續「暫停成長」而非續增)。
- 另外**把「連續」改名為「誠實回報連續」**,避免家長誤讀。

### B.（對應 2.3)48 小時自動放行,要補一刀維持信任真實性

自動放行防「打卡沒人理」死法,對。但若家長全程不看 → 全自動過 → 驗證形同虛設。

**我的提案(見 Q2):自動放行的打卡「照給金幣/XP(不懲罰小孩),但不算入信任升級」**——不能靠自動過畢業。這樣既不因家長缺席而處罰孩子,又保住信任系統的真實性。

### C. 小點

- 文件寫「後端定案 Supabase」,與我們前一輪講的 Firebase 相反。兩者都延後、都相容於 append-only 事件模型,只是請你確認最終是 Supabase(Q5)。
- **2.1 與 2.4 之間有個要釐清的軸線**:2.1 說任務用「銀段解鎖」(段位/XP),但 2.4 把信任從段位脫鉤。所以**任務解鎖是掛在「段位(XP 累積)」,信任是另一條「每行為驗證」軸**——這兩條刻意不同,對嗎?(見 Q7)
